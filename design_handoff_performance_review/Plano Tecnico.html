Especificação de arquitetura para a plataforma de avaliações de desempenho dos terceirizados da LeCode — modelo de dados, autorização por perfil e regras anti-viés impostas no servidor.

TypeScriptNext.js · App RouterReact TailwindCSSZustandSupabase · Postgres + RLS

00Visão & stack

O sistema avalia contratados da LeCode em ciclos periódicos. Cada ciclo coleta duas avaliações por contratado — uma autoavaliação (peso 30%) e uma avaliação do cliente (peso 70%) — sobre 5 dimensões, escala 1–5. O score final orienta decisões de promoção, PDI e recuperação.

A premissa central de segurança: avaliador e avaliado não podem ver a avaliação um do outro antes do encerramento do ciclo (anti-viés). Isso não é uma regra de UI — é imposto no banco via Row Level Security (RLS).

Princípio: o front é conveniência; toda decisão de acesso é reavaliada no Postgres. Um request direto à API REST/RPC do Supabase, sem passar pela UI, obedece exatamente às mesmas regras.

01Identidade & autorização

O perfil (lecode_admin, client_rep, contractor) é um atributo da conta, atribuído por um administrador — nunca escolhido pelo usuário no login ou no cadastro. Auto-seleção de perfil seria escalonamento de privilégio.

Fonte de verdade

A role vive em profiles.role (tabela 1:1 com auth.users). Novos cadastros entram como contractor; elevação para client_rep/lecode_admin é uma ação administrativa.

JWT como atalho de UI, não como autoridade

Espelhamos a role num custom claim (app_metadata.role) para o front rotear sem um round-trip. Mas o gate real é a RLS, que lê a role direto da tabela via auth.uid() — o claim nunca libera dados por si só.

supabase/functions · set_role_claim (Auth Hook)-- Custom Access Token Hook: injeta a role no JWT a cada emissão.
-- O valor vem SEMPRE do banco; o cliente não influencia.
create or replace function add_role_to_jwt(event jsonb)
returns jsonb language plpgsql as $$
declare r text;
begin
  select role::text into r from profiles
   where id = (event->>'user_id')::uuid;
  return jsonb_set(event, '{claims,app_metadata,role}', to_jsonb(r));
end; $$;

Auth0 vs Supabase Auth

Para esta stack, Supabase Auth é a escolha natural: Auth + Postgres + RLS integrados, e auth.uid() alimenta as políticas diretamente. Auth0 funciona, mas adiciona sincronização de usuário/role para o Postgres e configuração para o Supabase aceitar o JWT de terceiros (JWKS). Recomendado apenas se já houver Auth0 corporativo/SSO como requisito.

  submit_ends_on date not null, -- dia 15: fim do envio
  ends_on date not null,                 -- fim do mês: apuração
  status cycle_status not null default 'open',
  closed_at timestamptz, created_by uuid references profiles(id)
);

create table form_versions (
  id uuid primary key default gen_random_uuid(), version int not null,
  self_weight numeric not null default 0.30,
  client_weight numeric not null default 0.70,
  is_active boolean not null default false,
  check (self_weight + client_weight = 1)
);
create table form_questions (
  id uuid primary key default gen_random_uuid(),
  form_version_id uuid not null references form_versions(id) on delete cascade,
  dimension dimension_key not null, position int not null, text text not null
);

create table reviews (
  id uuid primary key default gen_random_uuid(),
  cycle_id uuid not null references cycles(id) on delete cascade,
  contractor_id uuid not null references contractors(id) on delete cascade,
  type review_type not null,
  author_id uuid not null references profiles(id),
  status review_status not null default 'draft',
  comments jsonb not null default '{}',   -- {strengths,growth,extra}
  submitted_at timestamptz, updated_at timestamptz default now(),
  unique(cycle_id, contractor_id, type)
);
create table review_answers (
  review_id uuid references reviews(id) on delete cascade,
  question_id uuid references form_questions(id),
  score smallint not null check (score between 1 and 5),
  primary key (review_id, question_id)
);

04Cálculo do score

As notas por dimensão e o score final são derivados (nunca gravados duplicados), expostos por views que respeitam a RLS (security_invoker). O peso vem da versão de formulário ativa.

supabase/migrations · 0002_scores.sqlcreate view review_dimension_scores with (security_invoker = on) as
select r.cycle_id, r.contractor_id, r.type, q.dimension,
       avg(a.score)::numeric(4,2) as dim_score
from reviews r
join review_answers a on a.review_id = r.id
join form_questions q on q.id = a.question_id
where r.status = 'submitted'
group by r.cycle_id, r.contractor_id, r.type, q.dimension;

-- Score final = self·0.30 + cliente·0.70 (pesos da versão ativa)
create view final_scores with (security_invoker = on) as
with overall as (
  select cycle_id, contractor_id, type, avg(dim_score) as s
  from review_dimension_scores group by 1,2,3
)
select o.cycle_id, o.contractor_id,
  max(o.s) filter (where type='self')   as self_score,
  max(o.s) filter (where type='client') as client_score,
  round(
    coalesce(max(o.s) filter(where type='self'),0)*fv.self_weight +
    coalesce(max(o.s) filter(where type='client'),0)*fv.client_weight, 2
  ) as final_score
from overall o
cross join (select self_weight, client_weight from form_versions where is_active) fv
group by o.cycle_id, o.contractor_id, fv.self_weight, fv.client_weight;
!
Como a view usa security_invoker, antes do ciclo fechar o avaliador não enxerga a linha da contraparte e o final_score sairia incompleto. Por isso o score final consolidado é exposto a não-admins por RPC com checagem explícita de ciclo encerrado (ver §07), e não direto pela view.

05Políticas RLS

Helpers stable centralizam a leitura do perfil; as policies traduzem cada regra de negócio. Toda tabela tem enable row level security e nenhuma policy permissiva por padrão.

0003_rls.sql · helperscreate function auth_role() returns user_role language sql stable as
$$ select role from profiles where id = auth.uid() $$;

create function is_admin() returns boolean language sql stable as
$$ select auth_role() = 'lecode_admin' $$;

create function my_client_id() returns uuid language sql stable as
$$ select client_id from profiles where id = auth.uid() $$;

create function my_contractor_id() returns uuid language sql stable as
$$ select id from contractors where profile_id = auth.uid() $$;

Reviews — leitura (anti-viés)

O coração das regras: o contratado vê a review do cliente só após o encerramento; o representante vê a autoavaliação só após o encerramento. Admin vê tudo.

0003_rls.sql · reviews (select)alter table reviews enable row level security;

-- Admin LeCode: acesso total
create policy reviews_admin_all on reviews
  for all using (is_admin()) with check (is_admin());

-- Contratado: sua self sempre; a do cliente só com ciclo fechado
create policy reviews_contractor_read on reviews for select using (
  contractor_id = my_contractor_id() and (
    type = 'self'
    or (type = 'client' and exists (
         select 1 from cycles c
         where c.id = reviews.cycle_id and c.status = 'closed'))
  )
);

-- Representante do cliente: contratados alocados no SEU cliente,
-- vínculo vigente no período do ciclo; a self só com ciclo fechado
create policy reviews_client_read on reviews for select using (
  auth_role() = 'client_rep'
  and exists (
    select 1 from allocations al
    join cycles c on c.id = reviews.cycle_id
    where al.contractor_id = reviews.contractor_id
      and al.client_id = my_client_id()
      and al.started_on <= c.ends_on
      and coalesce(al.ended_on, 'infinity') >= c.starts_on
  )
  and (
    type = 'client'
    or (type = 'self' and exists (
         select 1 from cycles c
         where c.id = reviews.cycle_id and c.status = 'closed'))
  )
);

Reviews — escrita (editar enquanto aberto)

Cada lado só escreve a própria review, e somente enquanto o ciclo estiver aberto — o que habilita naturalmente o "repensar a resposta" antes do encerramento e trava tudo depois.

0003_rls.sql · reviews (insert/update)-- Contratado cria/edita a própria self enquanto aberto
create policy reviews_self_write on reviews for insert with check (
  type = 'self' and contractor_id = my_contractor_id()
  and author_id = auth.uid()
  and exists(select 1 from cycles c where c.id = cycle_id and c.status='open')
);
create policy reviews_self_update on reviews for update
  using (type='self' and contractor_id = my_contractor_id()
     and exists(select 1 from cycles c where c.id=cycle_id and c.status='open'))
  with check (type='self' and contractor_id = my_contractor_id());

-- Representante cria/edita a review do cliente enquanto aberto,
-- exigindo vínculo ATIVO do contratado no seu cliente
create policy reviews_client_write on reviews for insert with check (
  type='client' and auth_role()='client_rep' and author_id = auth.uid()
  and exists(select 1 from cycles c where c.id=cycle_id and c.status='open')
  and exists(select 1 from allocations al
        where al.contractor_id = reviews.contractor_id
          and al.client_id = my_client_id() and al.ended_on is null)
);

review_answers herda a visibilidade da review-pai: a policy faz exists (select 1 from reviews r where r.id = review_id and <mesma lógica>), evitando duplicar regras.

07Ações sensíveis via RPC

Operações que exigem invariantes que uma policy de linha não expressa bem ficam em funções security definer, com checagens explícitas e search_path fixo.

0004_rpc.sql · close_cyclecreate or replace function close_cycle(p_cycle uuid)
returns void language plpgsql security definer set search_path = public as $$
begin
  if not is_admin() then raise exception 'forbidden'; end if;

  -- toda dupla (self, client) dos contratados alocados precisa estar submitted
  if exists (
    select 1 from allocations al
    join cycles c on c.id = p_cycle
    cross join (values ('self'),('client')) as t(type)
    where al.client_id is not null
      and al.started_on <= c.ends_on
      and coalesce(al.ended_on,'infinity') >= c.starts_on
      and not exists (
        select 1 from reviews r
        where r.cycle_id=p_cycle and r.contractor_id=al.contractor_id
          and r.type=t.type::review_type and r.status='submitted')
  ) then raise exception 'avaliacoes_pendentes'; end if;

  update cycles set status='closed', closed_at=now() where id=p_cycle;
end; $$;

get_final_score(p_cycle, p_contractor) segue o mesmo padrão: admin sempre; demais perfis só recebem o número se cycles.status = 'closed', evitando o vazamento indireto descrito na §04.

08Rotas Next.js (App Router)

O middleware renova a sessão e faz o gating grosseiro (autenticado? prefixo de role permitido?). O layout de cada área valida a role no servidor e os Server Components buscam dados com o JWT do usuário, de modo que a RLS é a barreira real.

├─ (auth)/                 // público
│  ├─ login/ · signup/ · recover/
├─ (app)/
│  ├─ layout.tsx           // getSession → profile.role; sem sessão → /login
│  ├─ admin/               // guard: role === 'lecode_admin'
│  │  ├─ page.tsx                 // visão geral
│  │  ├─ cycles/ · contractors/ · contractors/[id]/
│  │  ├─ clients/ · form/
│  ├─ client/              // guard: role === 'client_rep'
│  │  ├─ team/ · team/[contractorId]/evaluate/ · history/
│  └─ contractor/          // guard: role === 'contractor'
│     ├─ page.tsx · self-review/ · history/
└─ middleware.ts           // refresh de sessão + gate por prefixo de role
app/(app)/layout.tsxexport default async function AppLayout({ children }) {
  const supabase = createServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');

  // role lida do banco no servidor — não confiamos no claim do cliente
  const { data: profile } = await supabase
    .from('profiles').select('role, full_name').single();

  return <RoleProvider value={profile.role}>{children}</RoleProvider>;
}

09Stores Zustand

Zustand guarda estado de UI efêmero, não dados sensíveis nem a role como autoridade. O caso mais útil é o rascunho do formulário com autosave (que vira upsert de review draft enquanto o ciclo está aberto).

stores/reviewDraft.tstype DraftState = {
  answers: Record<string, number>;       // question_id → 1..5
  comments: { strengths: string; growth: string; extra: string };
  setAnswer: (q: string, v: number) => void;
  hydrate: (r: ReviewDraft) => void;       // preenche ao reabrir (editar)
  reset: () => void;
};

export const useReviewDraft = create<DraftState>((set) => ({
  answers: {}, comments: { strengths:'', growth:'', extra:'' },
  setAnswer: (q, v) => set((s) => ({ answers: { ...s.answers, [q]: v } })),
  hydrate: (r) => set({ answers: r.answers, comments: r.comments }),
  reset: () => set({ answers: {}, comments:{ strengths:'',growth:'',extra:'' } }),
}));

10Questões em aberto

  • Representante multi-cliente: profiles.client_id assume 1 cliente por rep. Se um rep cobre várias contas, virar tabela client_rep_assignments (N:N) e ajustar my_client_id() → my_client_ids().
  • Troca de cliente no meio do ciclo: as policies usam vigência por período; decidir se a avaliação "segue" o contratado ou fica congelada no cliente do início do ciclo.
  • Reabertura de ciclo: hoje encerrar é definitivo. Se precisar reabrir, exigir RPC admin + trilha de auditoria.
  • Auditoria & notificações: tabela audit_log (quem submeteu/editou/encerrou) e e-mails de abertura/lembrete/encerramento (Supabase + Edge Functions/Resend).
  • Histórico de pesos: ciclos antigos devem usar a versão de formulário vigente à época — vincular cycles.form_version_id em vez de sempre a versão ativa.
  • Rascunho vs enviado: confirmar se a "média parcial" some rascunhos ou só enviados nas views de score.
